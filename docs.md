# Noridoc: nori-browser

Path: @/

### Overview

- Electron desktop app that pairs a tabbed Chromium browser with a terminal sidebar, letting an AI agent (or human) control the browser via Playwright's CDP protocol
- The main process (`main.js`) orchestrates a multi-tab `WebContentsView` system, a `node-pty` terminal, an Electron `Menu` with keyboard accelerators, Chrome profile loading, omnibar autocomplete, and IPC handlers for the renderer toolbar and tab bar
- `playwright-bridge.js` is a stateless CLI tool that agents invoke from the terminal to read and manipulate browser pages and tabs

### How it fits into the larger codebase

- This is the top-level application directory; `@/renderer` contains the frontend UI and `@/test` contains end-to-end Playwright tests
- The terminal spawns whatever shell `NORI_BROWSER_SHELL` specifies, falling back to the **nori CLI driving the `claude-code` agent** (launched in an isolated `NORI_HOME` with the browser instructions injected as nori's initial prompt), then the user's default shell (see `resolveShell()` and `resolveNoriBinary()` in `main.js`). This fulfills the app's original intent in `APPLICATION-SPEC.md` ("claude code through the nori cli") and depends on the `nori-ai-cli` npm package
- The terminal environment is seeded with env vars that form the agent-to-browser contract:

| Env Var | Purpose |
|---|---|
| `NORI_BROWSER_CDP_PORT` | Port Electron exposes for Chrome DevTools Protocol |
| `NORI_BROWSER_HEADLESS` | When set, the BrowserWindow is created with `show: false` (no visible window). Used by tests to prevent Electron windows from appearing on screen |
| `PLAYWRIGHT_CDP_URL` | Full CDP URL (`http://localhost:<port>`) |
| `NODE_PATH` | Points to project `node_modules` so `require('playwright')` resolves from any cwd |
| `NORI_BROWSER_DIR` | Project root, so agent can locate `playwright-bridge.js` |
| `NORI_HOME` | Isolated nori config home (`~/.nori-browser`) seeded by `ensureNoriHome()`; keeps the browser session out of the user's personal `~/.nori/cli` config and disables nori's auto-worktree |
| `NORI_BROWSER_PROFILE_DIR` | Override for Chrome profile directory (empty string = use Electron default, unset = auto-detect Chrome) |
| `NORI_BROWSER_NORI_BIN` | Test seam: overrides the resolved nori binary path (used by the spawn-contract test to point at a stub nori) |

- The renderer communicates with the main process exclusively through IPC channels defined in `preload.js` -- no direct Node access from the renderer
- `playwright-bridge.js` connects over CDP, acts, and disconnects on every invocation -- it holds no persistent state or connections

### Core Implementation

- **Chrome profile loading**: Before `app.whenReady()`, `resolveProfileDir()` determines the user data directory. Priority: (1) `NORI_BROWSER_PROFILE_DIR` env var if set (empty string means "use Electron default"), (2) auto-detected Chrome user data dir via `getChromeUserDataDir()` (platform-specific paths), (3) null (Electron default). If Chrome is currently running (detected via `SingletonLock` symlink and `process.kill(pid, 0)` liveness check), the app falls back to Electron's default to avoid locking conflicts. When a profile dir is resolved, `app.setPath('userData', profileDir)` is called before `app.whenReady()` -- this is a hard requirement because Electron's path system locks after the ready event
- **History DB caching**: If a Chrome profile is loaded, the History SQLite database is copied to a temp file (`/tmp/nori-history-<pid>.db`) along with its WAL file before Electron takes ownership of the profile. This avoids Chromium's own schema migrations corrupting the read-only query path. The cached copy is cleaned up in the `window-all-closed` handler
- **Omnibar autocomplete**: The `omnibar-query` IPC handler (invoke/handle pattern) searches both Chrome Bookmarks (JSON file, recursive tree walk via `searchBookmarks()`) and History (SQLite via `better-sqlite3`). The SQL query filters and pre-sorts by `typed_count DESC, visit_count DESC, last_visit_time DESC` (LIMIT 200), but final ranking is done in JavaScript using a composite score: `prefixBoost + (typed_count * 4) + (visit_count * 0.5) + recencyScore`. Bookmarks get a fixed baseline score (20) plus any prefix boost. The `stripDomainPrefix()` helper strips `www.`, `m.`, and `mobile.` from domains before prefix matching so that e.g. `www.messenger.com` matches the query "mes". Bookmarks are searched first; their URLs are deduplicated against history results. Results are sorted by composite score descending and capped at 8 items. The renderer displays results in both an inline dropdown (for testability) and a native popup window (for display), with keyboard navigation, debounced input (150ms), and blur-to-close
- **Tab system**: Each tab is a `WebContentsView` with its own `webContents` and process, managed via module-level `tabs` array, `activeTabId`, and `nextTabId` counter. Each tab object has `{ id, view, title, url, pinned, favicon, isLoading }` where `pinned` defaults to `false`, `favicon` defaults to `''`, and `isLoading` defaults to `false`. `createTab()` creates a view, adds it to `mainWindow.contentView`, wires navigation/title/favicon/loading events, and switches to it. `createTab()` accepts an optional `insertIndex` parameter for position-controlled insertion (used by reopen and duplicate). Tab switching uses `view.setVisible(true/false)` rather than add/remove to avoid flicker. Closing the last tab closes the window
- **Closed tab stack**: A module-level `closedTabStack` (LIFO, capped at `MAX_CLOSED_TABS` = 25) stores `{ url, title, index }` for each closed tab. `reopenClosedTab()` pops the most recent entry and calls `createTab()` at the saved position. The stack is session-scoped only -- cleared on app exit, not persisted
- **Tab pinning**: `pinTab(tabId)` sets `pinned = true` and splices the tab to the end of the pinned zone (just before the first unpinned tab). `unpinTab(tabId)` sets `pinned = false` and moves the tab to the first unpinned position. This maintains the ordering invariant: all pinned tabs always precede all unpinned tabs in the `tabs` array. Pinned tabs are preserved by `closeOtherTabs()` and `closeTabsToRight()` (which filter them out before closing), but can still be closed individually via `closeTab()` (Ctrl+W, middle-click). Pinned state is included in `sendTabsChanged()` and `get-tabs` serialization so the renderer can style them
- **Tab batch operations**: `duplicateTab(tabId)` creates a new tab with the same URL inserted after the source tab (duplicates of pinned tabs are unpinned since `createTab()` defaults `pinned: false`). `closeOtherTabs(tabId)` and `closeTabsToRight(tabId)` close multiple tabs in a single operation, skipping pinned tabs. These are exposed as IPC channels and also available from the right-click tab context menu
- **Tab context menu**: When the renderer sends a `tab-context-menu` IPC with a `tabId`, the main process builds a native `Menu` via `Menu.buildFromTemplate()` and shows it with `menu.popup()`. The menu offers: New Tab, Reload, Duplicate, Pin Tab / Unpin Tab (label toggles based on current pin state), Close Tab, Close Other Tabs, Close Tabs to the Right
- **Application lifecycle**: `app.whenReady()` calls `createWindow()`, which triggers `startTerminal()` via IPC. The first tab (loading `about:blank`) is created in the `did-finish-load` handler. URL navigation is driven either by the toolbar URL bar (through `resolveInput()` for omnibox-style URL-vs-search classification) or by an agent calling the bridge CLI
- **Layout**: Two height constants control the browser area -- `TOOLBAR_HEIGHT` (48px) for the URL bar and `TAB_BAR_HEIGHT` (36px) for the tab strip. `setTabBounds()` positions each `WebContentsView` below both bars and to the right of the sidebar. `updateAllTabBounds()` is called on window resize, maximize, and sidebar resize events. When `isHtmlFullScreen` is true, `setTabBounds()` expands the view to fill the entire content area (x: 0, y: 0, full width/height), bypassing the normal toolbar/tab bar offsets
- **Menu structure**: The application menu has three top-level entries: **File** (New Tab, Close Tab, Reopen Closed Tab, Print), **View** (Zoom In/Out/Reset, Find in Page, Toggle Full Screen, Toggle Developer Tools), and **Tab** (Next/Previous Tab, Move Tab Right/Left, Tab 1-8, Last Tab). Many View menu items have corresponding keyboard shortcuts intercepted via `before-input-event` on each tab's `webContents` (F12/Ctrl+Shift+I for DevTools, Ctrl+F for find, Ctrl+=/Ctrl+-/Ctrl+0 for zoom, F11 for fullscreen, Ctrl+P for print)
- **Context menu (right-click in web content)**: Each `WebContentsView` has a `context-menu` event listener that builds a native menu dynamically based on what was clicked. Sections appear conditionally: links (Open Link in New Tab, Copy Link Address), images (Save Image As, Copy Image, Copy Image Address), selected text (Copy, Search Google for "..."), editable fields (Undo, Redo, Cut, Copy, Paste, Select All). Back/Forward/Reload/Inspect Element always appear at the bottom
- **Find-in-page**: Triggered by Ctrl+F or the View menu. The main process sends `show-find-bar` to the renderer, which displays a floating overlay. User input is forwarded to `webContents.findInPage()` via the `find-in-page` IPC channel. Match results come back through the `found-in-page` webContents event. `find-next` IPC advances matches (forward or backward). `find-close` calls `webContents.stopFindInPage('clearSelection')` and clears the stored `lastFindText`
- **Zoom**: A fixed array of Chrome-standard zoom factors (`ZOOM_FACTORS`, from 0.25 to 5.0) is defined at module level. `zoomIn()`/`zoomOut()` find the next/previous factor relative to the current level using a 0.001 epsilon for float comparison. `resetZoom()` sets the factor back to 1.0. Zoom changes are sent to the renderer via `zoom-changed` IPC (as percentage integers) for a transient overlay indicator
- **Downloads**: `session.defaultSession.on('will-download')` is registered inside `createWindow()`. Each download gets a monotonically increasing ID and is tracked in a module-level `downloads` Map. Progress updates (`updated` event) and completion (`done` event) are forwarded to the renderer via `download-progress` and `download-done` IPC. The renderer shows a download shelf with cancel/open/show actions that invoke `download-cancel`, `download-open`, `download-show` IPC channels back to main
- **Permissions**: `session.defaultSession.setPermissionRequestHandler` and `setPermissionCheckHandler` are configured inside `createWindow()`. Policy is deny-by-default, with explicit allowances for `fullscreen` and `clipboard-sanitized-write` only
- **HTML5 fullscreen**: `enter-html-full-screen` and `leave-html-full-screen` events on each tab's `webContents` toggle the `isHtmlFullScreen` flag, send `fullscreen-changed` IPC to the renderer (which hides/shows the toolbar and tab bar), and update the view bounds. Window-level fullscreen (F11) is separate and uses `mainWindow.setFullScreen()`
- **Status bar**: `update-target-url` events on each tab's `webContents` are forwarded to the renderer as `status-bar-url` IPC. The renderer shows/hides a bottom-left overlay displaying the hovered link URL
- **beforeunload handling**: Each tab's `webContents` listens for `will-prevent-unload` and shows a synchronous dialog ("Leave" / "Stay") via `dialog.showMessageBoxSync()`. Choosing "Leave" calls `event.preventDefault()` to allow navigation
- **Terminal agent launch (nori driving claude-code)**: `resolveShell(prompt, cwd)` resolves the launch command. `NORI_BROWSER_SHELL` (highest priority, used by tests) short-circuits to a bare shell. Otherwise `resolveNoriBinary()` finds the nori binary in order: `NORI_BROWSER_NORI_BIN` env (test seam) -> `node_modules/.bin/nori` (the installed `nori-ai-cli` dependency) -> `which nori`. When found, nori is invoked with `-a claude-code` (delegate each turn to the claude-code ACP backend), `-C <cwd>` (operate in the launch folder), `--dangerously-bypass-approvals-and-sandbox`, `--skip-welcome`, `--skip-trust-directory` (no blocking first-run prompts), `-c shell_environment_policy.inherit=all` (pass the seeded browser env vars through to the agent's shell), and the browser instructions string as the trailing **initial-prompt positional argument**. If no nori binary is found it falls back to `$SHELL`/`bash`
- **Browser instructions delivery**: `buildBrowserPrompt()` returns the instruction text (no temp file is written). It tells the agent how to connect to the browser (CDP port, `playwright-bridge.js` path, available commands), documents the `window.api` tab operations available via the renderer (e.g., `duplicateTab`, `closeOtherTabs`, `reopenClosedTab`), instructs it not to use MCP tools, create worktrees, or run `git init`, and includes network etiquette rules (random inter-request delays, exponential backoff with jitter, 429/Retry-After honoring, concurrency limits, robots.txt checks) that apply only to external services (not localhost/private networks). The instructions are delivered as nori's **initial user turn** rather than via config or a system-prompt file, because nori forwards only the initial user turn to the claude-code ACP backend — config-based `base_instructions`/`user_instructions` are not propagated to the ACP agent
- **Isolated nori home**: `ensureNoriHome()` creates `~/.nori-browser` containing a `config.toml` with `[tui] auto_worktree = "off"`, and points the pty's `NORI_HOME` env var at it. This keeps the browser session out of the user's personal `~/.nori/cli` config (mirroring the isolation goal of the old claude `--setting-sources ''`) **and** disables nori's auto-worktree feature — without this, nori would pull the agent into a separate git worktree instead of the folder the browser launched from and block on a "Worktree cannot be created" modal at startup
- **Browser-terminal integration flow**:
```
Agent types in terminal
  -> pty forwards to shell
  -> shell runs `node $NORI_BROWSER_DIR/playwright-bridge.js <cmd>`
  -> bridge connects via CDP to Electron's WebContentsView tabs
  -> bridge performs action, prints status marker (e.g. NAVIGATE_OK), disconnects
  -> terminal shows output to agent
```
- **Tab favicon & loading indicators**: Each tab's `webContents` listens for `page-favicon-updated` (stores the first favicon URL), `did-start-loading` / `did-stop-loading` (toggles `isLoading`), and `did-start-navigation` (clears `favicon` on main-frame cross-document navigations to avoid showing stale favicons). These fields are included in `sendTabsChanged()` and `get-tabs` serialization so the renderer can display favicons and loading spinners
- **IPC channels** in `preload.js` expose a `window.api` surface organized into groups: terminal I/O, navigation, layout, state sync, tab management, omnibar (`omnibar-query` via invoke/handle), find-in-page (`findInPage`, `findNext`, `findClose`, `onFindResults`, `onShowFindBar`), zoom (`onZoomChanged`), downloads (`onDownloadProgress`, `onDownloadDone`, `downloadCancel`, `downloadOpen`, `downloadShow`), status bar (`onStatusBarUrl`), and fullscreen (`onFullscreenChanged`)
- **Sidebar toggle (Ctrl+J)**: The main process owns the toggle state (`sidebarVisible`, `savedSidebarWidth`). Pressing Ctrl+J triggers `handleToggleSidebar()`, which saves/restores the sidebar width, recalculates all tab view bounds, and notifies the renderer via `sidebar-toggled` IPC. The shortcut is intercepted at multiple levels: `before-input-event` on `mainWindow.webContents` and each tab's `webContents`, plus a renderer-side `keydown` listener and xterm custom key handler (see `@/renderer/renderer.js`)
- **Bridge CLI** (`playwright-bridge.js`) supports single-page commands (`status`, `navigate`, `snapshot`, `click`, `fill`, `eval`, `content`, `screenshot`) and tab management commands (`list-tabs`, `new-tab`, `close-tab`, `switch-tab`). It also exports `connectToBrowser()` for programmatic use
- **Build step**: `renderer/renderer.js` is bundled via esbuild into `renderer/bundle.js` before the app starts or tests run
- **`window.open()` interception**: Each tab's `webContents` has a `setWindowOpenHandler` that intercepts `window.open()` calls and opens them as new tabs instead of new windows

### Things to Know

- The bridge CLI finds the correct page by filtering out `file://` URLs (the renderer's `index.html`) and picking the first `http`-prefixed page. If no HTTP page exists, it falls back to `pages[0]`
- `playwright` is a runtime dependency (not dev-only) because the bridge CLI needs it when invoked from the terminal. `better-sqlite3` is a native module rebuilt for Electron via `@electron/rebuild` -- it provides synchronous SQLite access for the omnibar history queries
- The CDP port defaults to `19222` but is configurable via `NORI_BROWSER_CDP_PORT`. Each test suite uses a distinct port offset (e.g., `19223`, `19233`, `19243`, etc.) to avoid collisions with each other and with a running dev instance
- URL bar input goes through `resolveInput()` in `main.js`, which classifies text as either a URL or a search query. URLs (detected by explicit scheme, dots, IP addresses, `localhost`, or port patterns) get navigated directly with an appropriate protocol prefix. Everything else becomes a Google search via `https://www.google.com/search?q=<encoded_query>`. The classification uses a priority-ordered heuristic chain -- whitespace triggers search early, dots trigger URL treatment, and single words without dots fall through to search. Google is hardcoded as the search engine
- Profile detection runs synchronously at module load time (before `app.whenReady()`) because `app.setPath()` must be called before the app ready event. This means `getChromeUserDataDir()`, `isChromeRunning()`, and `resolveProfileDir()` are all synchronous. If Chrome is running concurrently, Nori falls back gracefully with a stderr warning
- The `isChromeRunning()` check uses Linux's `SingletonLock` symlink convention -- it reads the symlink target, extracts the PID, and sends signal 0 to verify the process is alive. This is Linux-specific; on other platforms the function will return `false` (no lock file found), which may cause profile conflicts if Chrome is actually running
- The omnibar query path reads from `cachedHistoryPath` (temp copy) when available, falling back to the live `History` file inside `Default/`. The `better-sqlite3` connection is opened in `readonly` mode with `fileMustExist: true`, and closed immediately after each query -- no persistent DB handles
- `resolveNoriBinary()` falls back to a synchronous `execSync('which nori')` only after checking `NORI_BROWSER_NORI_BIN` and the bundled `node_modules/.bin/nori` -- if no nori binary is found anywhere, `resolveShell()` silently falls back to the default shell. The browser instructions are passed to `resolveShell()` as a string argument and become nori's initial-prompt positional
- On Linux, the `maximize` and `unmaximize` window events fire before the window manager has updated `mainWindow.getContentSize()`. A single bounds update call reads stale dimensions. The workaround is a deferred second call via `setTimeout(fn, 100)` that runs after the WM has settled. The `resize` event (manual drag) does not need this because the WM updates dimensions synchronously for drag operations
- Each bridge CLI invocation opens and closes a full CDP connection. This is intentional -- statelessness means no zombie connections, but it adds latency per command
- At startup nori emits a Device Status Report query (`\x1b[6n`) and exits within ~2s if nothing answers; the pty `onData` handler watches for this and auto-replies with a cursor-position report (`\x1b[1;1R`) so nori stays alive even when no real terminal is attached
- The isolated `NORI_HOME` directory (`~/.nori-browser`) is created once by `ensureNoriHome()` and is **not** torn down on exit (it holds only a static `config.toml`). The cached history temp file is still cleaned up in `window-all-closed`. There is no longer a per-session temp directory or `before-quit` cleanup handler
- Tab IDs are stringified integers from a monotonically increasing counter (`nextTabId`). They are not reused, so a tab with ID "3" may exist while tabs "1" and "2" are already closed
- `reorderTab()` clamps target indices to respect the pinned/unpinned boundary: pinned tabs can only move within indices `[0, pinnedCount-1]`, unpinned tabs within `[pinnedCount, tabs.length-1]`. `moveTabLeft()`/`moveTabRight()` enforce the same boundary by refusing to move a tab across the pin zone edge
- Navigation events (`did-navigate`, `did-navigate-in-page`) and title events (`page-title-updated`) on each tab's `webContents` keep the tab metadata, URL bar, and window title in sync only for the active tab
- Session-level handlers (downloads, permissions) are initialized inside `createWindow()` rather than at module level, ensuring `session.defaultSession` is ready before registration
- Each `WebContentsView` tab has event listeners for: `context-menu`, `found-in-page`, `update-target-url`, `enter-html-full-screen`, `leave-html-full-screen`, `will-prevent-unload`, plus the pre-existing navigation, title, favicon, and loading events. All are registered in `createTab()`
- The `before-input-event` handler on each tab's `webContents` intercepts keyboard shortcuts that would otherwise be consumed by the web page: Ctrl+J (sidebar toggle), Ctrl+Shift+I / F12 (DevTools), Ctrl+F (find), Ctrl+=/Ctrl+-/Ctrl+0 (zoom), F11 (fullscreen), Ctrl+P (print). Each calls `event.preventDefault()` before taking action
- Download IDs, like tab IDs, are monotonically increasing stringified integers from `nextDownloadId`. The `downloads` Map is never cleaned up -- completed downloads remain in the Map for the lifetime of the session so that `download-open` and `download-show` IPC can still resolve save paths
