# Noridoc: nori-browser

Path: @/

### Overview

- Electron app that pairs a Chromium browser (right panel) with an AI-powered terminal (left sidebar), connected via CDP so terminal agents can control the browser
- Main process (`main.js`) orchestrates window lifecycle, PTY management, shell resolution, and IPC routing between renderer and PTY
- Preload script (`preload.js`) exposes a `window.api` bridge for the renderer to communicate with the main process over IPC

### How it fits into the larger codebase

- `main.js` is the Electron main process entry point; it creates the BrowserWindow, attaches a BrowserView for the Chromium panel, spawns a PTY for the terminal, and handles all IPC
- `preload.js` defines the `window.api` surface that `renderer/renderer.js` calls into -- every terminal and navigation action flows through this bridge
- The PTY process runs whichever shell `resolveShell()` finds (nori CLI > claude CLI > `$SHELL` > `/bin/bash`), with browser-context instructions injected as CLI args for AI shells
- The BrowserView exposes CDP on a configurable port (`NORI_BROWSER_CDP_PORT`, default 19222), which the PTY process receives as `PLAYWRIGHT_CDP_URL` in its environment
- `playwright-bridge.js` is a standalone utility for connecting to the browser via CDP -- it is not imported by the app itself but is available for terminal scripts
- Tests in `test/app.test.js` launch the full Electron app with Playwright and exercise the terminal, browser navigation, CDP connectivity, and ESC sequence handling

### Core Implementation

- **Startup sequence**: `app.whenReady()` calls `createWindow()`, then waits for a `terminal-ready` IPC signal from the renderer before calling `startTerminal()`. This ensures xterm.js is initialized and fitted before the PTY emits any output.

```
app.whenReady() -> createWindow() -> renderer loads -> xterm.js fit -> IPC 'terminal-ready' -> startTerminal()
```

- **Shell resolution** (`resolveShell()`): checks `NORI_BROWSER_SHELL` env var first, then `which nori`, then `which claude`, then falls back to `$SHELL` or `/bin/bash`. For nori and claude, it appends CLI-specific flags with a `BROWSER_CONTEXT` prompt explaining how to connect to the browser via CDP.
- **Session directories** (`createSessionDir()`): each app launch creates `~/nori-browser/YYYYMMDD-HHMMSS/` and sets it as the PTY's working directory. These directories accumulate and are not automatically cleaned up.
- **ESC[6n interception**: when the PTY output contains a cursor position query (`ESC[6n`, commonly sent by Rust CLIs using crossterm), the main process intercepts it, writes `ESC[1;1R` back to the PTY immediately, and strips the sequence from data forwarded to xterm.js. This avoids a timeout that would occur if the query had to round-trip through IPC to xterm.js and back.
- **IPC channels**: terminal data flows as `terminal-input` (renderer->main->PTY) and `terminal-data` (PTY->main->renderer). Navigation uses `navigate`, `go-back`, `go-forward`, `reload`. Layout uses `sidebar-resize`. Metadata uses `cdp-port` and `url-changed`.

### Things to Know

- The PTY spawn is intentionally deferred until the renderer signals readiness via `terminal-ready`. If this signal never arrives (e.g., renderer crash), the terminal never starts. There is no timeout fallback.
- The `ESC[6n` interception always responds with position `1;1` regardless of actual cursor position. This is a deliberate simplification -- the response just needs to be fast enough to satisfy crossterm's timeout, and the actual position value is rarely used by the caller.
- `resolveShell()` uses synchronous `execSync('which ...')` calls at startup. If these hang (e.g., broken PATH), the app will block.
- The `BROWSER_CONTEXT` string is a multi-line prompt injected differently per CLI: nori gets `-c developer_instructions=...`, claude gets `--append-system-prompt ...`. If neither CLI is found, no context is injected into the fallback shell.
- `renderer/bundle.js` is a build artifact generated by esbuild from `renderer/renderer.js` -- edit the source, not the bundle. Build with `npm run build`.
